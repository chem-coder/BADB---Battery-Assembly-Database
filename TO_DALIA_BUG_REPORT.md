# TO_DALIA_BUG_REPORT — Результаты аудита кода (апрель 2026)

Привет, Даля! Мы провели полный аудит кодовой базы и нашли несколько багов в серверных маршрутах и фронтенде. Баги в **нашем** коде (frontend-april-sync) мы уже исправили. Ниже — баги, которые касаются **твоего** кода и которые было бы хорошо поправить.

---

## Серверные баги (routes/)

### B8 — routes/users.js: POST /api/users не хеширует пароль
**Файл:** `routes/users.js`
**Суть:** Эндпоинт `POST /api/users` сохраняет пароль напрямую в `password_hash` без вызова `bcrypt.hash()`. Если через этот маршрут создают пользователя, пароль хранится в открытом виде.
**Рекомендация:** Добавить `const passwordHash = await bcrypt.hash(password, config.bcrypt.rounds)` перед INSERT, аналогично `routes/auth.js` → `/register`.

### B9 — routes/users.js: PUT обновляет пароль без хеширования
**Файл:** `routes/users.js`
**Суть:** При обновлении пользователя, если передан `password`, он записывается в `password_hash` без хеширования.
**Рекомендация:** Аналогично B8 — хешировать через bcrypt перед UPDATE.

### B10 — routes/electrodes.js: SQL-инъекция через интерполяцию строк
**Файл:** `routes/electrodes.js`
**Суть:** В `GET /api/electrodes` для построения условий `WHERE` используется конкатенация строк (например, подстановка `project_id` напрямую в SQL). Это открывает путь для SQL-инъекции.
**Рекомендация:** Перевести на параметризованные запросы (`$1, $2, ...`).

### B11 — routes/electrodes.js: DELETE без проверки существования
**Файл:** `routes/electrodes.js`
**Суть:** `DELETE /api/electrodes/:id` не проверяет `rowCount` после DELETE. Если электрод не найден, возвращается 200 (OK) вместо 404.
**Рекомендация:** Проверить `result.rowCount === 0` → вернуть 404.

### B12 — routes/tapes.js (Dalia's endpoints): GET /api/tapes/test без try/catch
**Файл:** `routes/tapes.js`
**Суть:** Эндпоинт `/test` не оборачивает запрос к БД в try/catch. При ошибке соединения Express вернёт 500 с stack trace.
**Примечание:** Мы уже обернули в try/catch в нашей ветке, но если ты будешь менять этот файл в main — учти.

### B13 — Несколько CRUD-маршрутов: PUT возвращает 200 даже если запись не найдена
**Файлы:** `routes/electrodes.js`, `routes/projects.js` и другие CRUD
**Суть:** После `UPDATE ... WHERE id = $1` не проверяется `rowCount`. Если записи нет — клиент получает 200 с пустым результатом.
**Рекомендация:** Проверять `result.rowCount === 0` → 404.

### B14 — routes/reference/*.js: Отсутствие валидации входных данных
**Файлы:** справочные маршруты (`routes/reference/`)
**Суть:** POST/PUT эндпоинты для справочников не валидируют обязательные поля. Можно создать запись с пустым именем.
**Рекомендация:** Добавить минимальную валидацию (`if (!name) return res.status(400)...`).

### B16 — Нет ротации JWT при смене пароля
**Файл:** `routes/auth.js`
**Суть:** После `PUT /api/auth/change-password` и `POST /api/auth/change-password-public` старый JWT остаётся валидным до истечения (8ч). Если пароль меняется из-за компрометации, атакующий сохраняет доступ.
**Рекомендация:** После смены пароля выпускать новый токен и/или инвалидировать старые (через `token_version` в таблице users + проверку в middleware).

## Фронтенд (public/)

### F17 — public/*.html: Захардкоженный порт 3000
**Файлы:** HTML-файлы в `public/`
**Суть:** Fetch-запросы в HTML-страницах используют `http://localhost:3000/api/...`. Сервер работает на порту 3003.
**Рекомендация:** Заменить на относительные URL (`/api/...`) — тогда запросы пойдут через текущий хост/порт.
